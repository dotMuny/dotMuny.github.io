# ADscan: Automating Active Directory Pentesting Like a Pro

> **TL;DR:** ADscan is an open-source pentesting tool that automates Active Directory reconnaissance, enumeration, and exploitation through an intelligent interactive CLI - cutting assessment time from hours to minutes.

---

## What is ADscan?

If you've ever spent a full day manually enumerating an Active Directory environment - running BloodHound, firing off Impacket scripts, chasing Kerberoastable accounts, and correlating it all by hand - you already understand the problem ADscan was built to solve.

**ADscan** (by [ADScanPro](https://www.adscanpro.com/)) is a professional-grade pentesting tool that wraps the most critical AD attack paths into a single, cohesive interactive shell. It covers the full lifecycle of an internal AD audit: unauthenticated reconnaissance, authenticated enumeration, Kerberos attacks, credential dumping, and BloodHound path analysis - all from one place.

It's written in Python, runs on Linux (Debian/Ubuntu/Kali), and is available as a free LITE edition with a PRO edition on the way.

---

## Why ADscan Matters

Active Directory remains the backbone of enterprise identity and access management. It's also the most consistently attacked surface in internal penetration tests. The challenge isn't knowing _what_ to do - it's doing it fast, systematically, and without missing critical misconfigurations.

ADscan addresses this in three ways:

**Speed.** What normally takes hours of manual chaining (nmap → ldapdomaindump → BloodHound → GetNPUsers.py → secretsdump.py) is compressed into a guided, semi-automatic workflow.

**Consistency.** Human error during audits is real. Forgetting to check AS-REP roasting or missing an ACL path is easy when you're juggling multiple tools. ADscan enforces a structured methodology.

**Accessibility.** Whether you're a seasoned red teamer or a junior analyst on your first internal engagement, ADscan's interactive CLI guides you through each step with prompts and autocomplete.

## Installation

ADscan offers three installation methods. The recommended approach is via `pipx`, which isolates the tool's dependencies cleanly.

### Option 1: pipx (Recommended)

```bash
pipx install adscan
```

### Option 2: pip

```bash
pip install adscan
```

### Option 3: Pre-built Binary

If you prefer a standalone binary without managing Python environments:

```bash
# Download the latest release binary
wget https://github.com/ADscanPro/adscan/releases/latest/download/adscan

# Make it executable
chmod +x adscan

# Move it to your PATH
sudo mv adscan /usr/local/bin/
```

### Post-Installation Setup

Regardless of the method used, run the following two commands after installation:

```bash
# Install Python environment, required tools, and wordlists
adscan install

# Verify everything is correctly set up
adscan check
```

The `adscan install` command pulls in the required dependencies and wordlists automatically. The `adscan check` command validates that all external tools (like BloodHound, Impacket, etc.) are accessible.

## Quick Start: Your First Scan

### Step 1 - Launch ADscan

```bash
adscan start -v
```

The `-v` flag enables verbose output, useful when you want to see what's happening under the hood.

You'll be dropped into an interactive shell prompt:

```
(ADscan) >
```

### Step 2 - Create a Workspace

Workspaces are isolated environments that store your target configuration, credentials, and scan results.

```bash
(ADscan) > workspace create my_audit
```

Your prompt updates to reflect the active workspace:

```
(ADscan:my_audit) >
```

### Step 3 - Configure Your Target

Set the network interface and scan mode:

```bash
(ADscan:my_audit) > set iface tun0
(ADscan:my_audit) > set auto False   # Semi-automatic mode (recommended for audits)
```

> **Semi-automatic mode (`auto=False`)** pauses before executing each attack phase and asks for confirmation. This is ideal for production environments or when you need to document each step. **Automatic mode (`auto=True`)** runs everything end-to-end without prompting - great for CTFs and labs.

## Running Scans

### Unauthenticated Scan

Use this when you have no credentials and need to enumerate the environment from scratch:

```bash
(ADscan:my_audit) > set hosts 192.168.1.0/24
(ADscan:my_audit) > start_unauth
```

ADscan will automatically discover domain controllers, identify domain names, attempt AS-REP roasting against accounts with Kerberos pre-authentication disabled, collect BloodHound data, and guide you through the enumeration chain.

### Authenticated Scan

Once you have valid credentials (even low-privilege ones), launch an authenticated scan for full enumeration:

```bash
(ADscan:my_audit) > start_auth domain.local 10.10.10.1 username password
```

This triggers a deeper phase: Kerberoasting, ACL analysis, BloodHound path collection, privilege escalation checks, and credential dumping opportunities.

### Supplying an NTLM Hash Instead of a Password

ADscan supports pass-the-hash style authentication:

```bash
(ADscan:my_audit) > start_auth domain.local 10.10.10.1 username :aad3b435b51404eeaad3b435b51404ee
```

## CI/CD Mode: Non-Interactive Automation

ADscan includes a `ci` subcommand for running scans in automated pipelines - think CI/CD, scheduled jobs, or integration with red team frameworks.

### Unauthenticated CI Scan

```bash
adscan ci unauth \
  --type ctf \
  --interface tun0 \
  --hosts 10.10.10.10
```

### Authenticated CI Scan

```bash
adscan ci auth \
  --type ctf \
  --interface tun0 \
  --domain example.local \
  --dc-ip 10.10.10.1 \
  --username user \
  --password pass
```

### Keeping the Workspace for Debugging

```bash
adscan ci unauth \
  --type ctf \
  --interface tun0 \
  --hosts 10.10.10.10 \
  --keep-workspace
```

**Exit codes** from CI mode are meaningful:

- `0` - Scan succeeded and flags/objectives were validated
- `1` - Scan failed
- `2` - Scan completed but flags were invalid or missing

This makes ADscan easy to integrate with automated lab validation systems like HTB Pro Labs or internal training environments.

## Workspace and Credential Management

ADscan stores all data locally under `~/.adscan/workspaces/`, organized by workspace name. This means:

- Each audit target gets its own isolated directory
- Credentials discovered during a scan are automatically stored and reused
- You can resume an audit later without losing context
- No sensitive data is transmitted externally (telemetry is opt-in and anonymized)

To list existing workspaces:

```bash
(ADscan) > workspace list
```

To switch between workspaces:

```bash
(ADscan) > workspace load previous_audit
```

---

## BloodHound Integration

ADscan natively collects BloodHound data during authenticated scans and can trigger automated path analysis. If you have BloodHound and Neo4j set up locally, the tool will feed data directly into the graph database and highlight the shortest attack paths to Domain Admin.

This removes the need to run BloodHound Collectors separately and manually import ZIP files - a small but meaningful time saver in a real engagement.

---

## Privacy and Telemetry

ADscan collects **opt-in, anonymized telemetry** covering error data and feature usage. No sensitive information (credentials, hostnames, scan results) is transmitted.

To disable telemetry entirely:

```bash
(ADscan:workspace) > set telemetry off
```

All scan data remains local, stored only in your workspace directory.

## Should You Use ADscan?

ADscan fits well in a few specific contexts:

**CTF players and students** - The automatic mode is genuinely impressive for lab environments. If you're working through AD-focused machines on HTB or similar platforms, it will cut your time significantly while showing you the attack chain clearly.

**Junior pentesters** - The semi-automatic mode acts almost like a guided methodology. Each phase is explained and prompted, which helps build intuition for AD attack paths.

**Experienced red teamers** - The value here is speed and consistency. You probably know all the techniques already, but ADscan removes the operational friction of juggling a dozen tools and manually tracking progress.

**Production environments** - Use semi-automatic mode. Never run `auto=True` against production AD without explicit permission and careful review of each step.
